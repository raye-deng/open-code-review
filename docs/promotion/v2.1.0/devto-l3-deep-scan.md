I built an open-source tool that catches AI code bugs that ESLint misses — now with LLM deep scan

Last month, a junior developer on my team committed code generated by Cursor that imported `@aws-sdk/client-s3-v3` — a package that has never existed. ESLint? Silent. TypeScript compiler? Silent. SonarQube? Silent. It passed all checks and landed in production.

That's when I realized something: the tools we trust to catch bad code were designed for humans, not AI assistants. And AI-generated code has an entirely different class of defects.

## The 5 AI Code Defects Linters Can't See

After scanning hundreds of thousands of lines of AI-generated code, I've identified five recurring defect patterns that traditional linting tools consistently miss:

### 1. Hallucinated Imports

AI models sometimes invent package names that sound plausible but don't exist on any registry:

```typescript
// Looks correct. Linting passes. But this package doesn't exist.
import { validateEmail } from 'validator-email-utils';
import { S3Client } from '@aws-sdk/client-s3-v3';
```

ESLint checks syntax, not registry existence. SonarQube checks patterns, not whether the package is real. This code will fail at runtime — or worse, at deploy time.

### 2. Stale APIs

AI models are trained on data with a cutoff date. They frequently suggest APIs that were deprecated years ago:

```python
# Deprecated in 2020. AI still recommends it.
import imp
module = imp.load_source('mymodule', '/path/to/file.py')

# Or in JavaScript:
const fs = require('fs');
fs.exists('/tmp/file', (exists) => { ... }); // Deprecated since Node 4
```

These work — until they don't. When the platform finally removes the deprecated API, your code breaks with zero warning.

### 3. Context Window Artifacts

When AI generates code across multiple files in a conversation, it can lose track of what it already wrote:

```typescript
// file-a.ts — AI generates this
export function getUser(id: string): Promise<User> {
  return db.query('SELECT * FROM users WHERE id = ?', [id]);
}

// file-b.ts — AI generates this 15 messages later
export function getUser(id: string): Promise<User> {
  return db.query('SELECT * FROM users WHERE user_id = ?', [id]);
}
```

Two functions with the same name, different implementations, different SQL queries. No linter catches this because syntactically, both files are perfectly valid.

### 4. Over-Engineering

AI loves patterns. Too much, sometimes:

```typescript
// AI generates a Strategy pattern for a simple toggle
interface FeatureToggleStrategy {
  isEnabled(feature: string): Promise<boolean>;
}

class DefaultFeatureToggleStrategy implements FeatureToggleStrategy {
  async isEnabled(feature: string): Promise<boolean> {
    return this.config.get(feature) ?? false;
  }
}

class FeatureToggleFactory {
  static create(config: Config): FeatureToggleStrategy {
    return new DefaultFeatureToggleStrategy(config);
  }
}
```

When a simple `config.get(feature) ?? false` would do. Over-engineered code is harder to maintain, test, and understand — but no linter flags it as "unnecessary complexity."

### 5. Security Anti-Patterns

AI models trained on blog posts and Stack Overflow often reproduce insecure patterns:

```javascript
// AI-generated authentication check
app.post('/login', (req, res) => {
  const { user, pass } = req.body;
  if (user === 'admin' && pass === 'password123') { // ⚠️
    res.json({ token: 'admin-token-12345' }); // ⚠️ Hardcoded
  }
});

// Or using eval() for JSON parsing
const data = eval(req.body.jsonString); // ⚠️ eval injection
```

SonarQube catches some of these, but AI-generated security issues often use novel combinations that don't match known rule patterns.

## The Architecture: A Three-Level SLA Pipeline

The core insight behind Open Code Review is that not all checks need the same amount of compute. I designed a three-level SLA pipeline:

**L1 Fast (<10 seconds, FREE, 100% local)**
- Structural analysis: import validation against npm/PyPI/Maven registries
- Pattern matching for known AI defect signatures
- Security anti-pattern detection
- Basic over-engineering heuristics

**L2 Standard (adds embedding recall + local LLM)**
- Everything in L1, plus
- Code embedding comparison to detect context window artifacts
- Local LLM (via Ollama) for semantic analysis
- Cross-file consistency checks

**L3 Deep (adds remote LLM deep scan)**
- Everything in L1+L2, plus
- Sends suspicious code blocks to a remote LLM for deep analysis
- Can reason about business logic correctness, API usage patterns, and architectural concerns
- The focus of v2.1.0

The key design principle: each level is additive. You never lose coverage by running L1 instead of L3 — you just trade depth for speed.

## How L3 Deep Scan Works

L3 is the most interesting part of v2.1.0. Here's the pipeline:

### Step 1: Identify Suspicious Blocks

After L1+L2 runs, we have a list of issues. But we also identify "suspicious" blocks — code that looks structurally fine but has red flags:

- Functions that are unusually long (>50 lines)
- Imports from packages not in the project's dependency tree
- Code patterns that don't match the project's existing style
- Complex conditional logic that could hide logical errors

```typescript
// Suspicious: function is 80 lines, handles 7 different cases,
// and imports from a package not in package.json
async function processOrder(order: Order, user: User, config: Config, db: Database): Promise<Result> {
  // ... 80 lines of nested conditionals
}
```

### Step 2: Send to Remote LLM

We don't send your entire codebase. We extract targeted blocks and ask the LLM specific questions:

```json
{
  "prompt": "Analyze this function for AI-generated code defects:\n1. Does it use deprecated APIs?\n2. Are there unnecessary abstractions?\n3. Are there security anti-patterns?\n4. Does the logic correctly handle edge cases?\n\nFunction:\n```typescript\nasync function processOrder(...) { ... }\n```",
  "max_tokens": 1000,
  "temperature": 0.1
}
```

### Step 3: Parse and Merge Results

The LLM response is parsed into structured findings and merged with L1+L2 results. Each finding includes severity, location, and a suggested fix.

## Universal Provider Adapter: Why 8 Providers?

One thing I didn't want was vendor lock-in. The Universal Provider Adapter abstracts LLM access behind a simple interface:

```typescript
interface LLMProvider {
  complete(prompt: string, options: ProviderOptions): Promise<string>;
}
```

Currently supported:

| Provider | Cost | Notes |
|----------|------|-------|
| **GLM/ZAI** | **Free** | Generous free tier, great for open-source |
| OpenAI | Pay-per-token | GPT-4o, GPT-4-turbo |
| Anthropic | Pay-per-token | Claude 3.5 Sonnet, Haiku |
| DeepSeek | Pay-per-token | Cost-effective, strong coding |
| Together AI | Pay-per-token | Open-source model hosting |
| Fireworks | Pay-per-token | Fast inference |
| Ollama | Free (local) | Run any model locally |
| OpenAI-compatible | Varies | Works with any compatible API |

The highlight: **GLM/ZAI offers a free tier** that's genuinely usable. You can run L3 deep scans on every CI build without paying a cent. This is a big deal for open-source projects and individual developers.

## v2.1.0 Also Brings: Multi-Language Support

AI code defects aren't limited to TypeScript. v2.1.0 adds dedicated detectors for six languages:

- 🟦 **TypeScript / JavaScript** — npm registry validation, Node.js API staleness
- 🐍 **Python** — PyPI registry validation, stdlib deprecation checks
- ☕ **Java** — Maven Central validation, Jakarta EE migration issues
- 🔵 **Go** — module path validation, stdlib API changes
- 🟣 **Kotlin** — Maven Central validation, Android API deprecation

Each language has dedicated detectors tuned for its ecosystem's specific failure modes.

jobs:
  review:
    runs-on: ubuntu-latest

      - uses: actions/checkout@v4
      - uses: raye-deng/open-code-review@v1
